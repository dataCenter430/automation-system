Authoritative distillation of the Snorkel/Terminus Edition-2 documentation set.
Read this end to end before you touch a file. It is the spec you will be graded against.

You are about to build ONE complete Terminus Edition-2 task from scratch. Everything you need

Nothing else in this document matters if you fail this gate. It is not a formatting rule and it

================================================================================
SECTION 1. THE MANDATE
================================================================================

Build a task that frontier models FAIL.

The acceptance bar, stated bluntly:

  - Target pass rate for GPT-5.5 and Claude Opus 4.8: 0% to 20%. That is the "hard" band.
  - A task whose WORST model scores above 80% is REJECTED outright. No appeal.
  - Pass rate must be below 100% against SOTA agents, full stop.
  - The ORACLE (solution/solve.sh) must pass: reward.txt == 1.
  - The NULL / NOP run (agent does nothing) must fail: reward.txt == 0.
  - Difficulty must be earned by genuine engineering depth, NOT by piling on instructions.
    A task that is hard only because it has 30 fiddly requirements is explicitly rejected
    ("If a task is primarily hard due to a large number of edge cases/requirements that are
    not handled well, it should be rejected"). Hard means the *problem* is hard.

Difficulty bands (from Task Requirements.txt — this is the authoritative table):

  Hard    accuracy <= 20% on the best model, OR <= 20% on the worst model
  Medium  20% < accuracy <= 60% on the worst model
  Easy    60% < accuracy <= 80% on the worst model
  >80% on the worst model  ->  NOT ACCEPTED

You are building HARD. Set difficulty = "hard" in task.toml and make the task deserve it.

Two invariants you will be independently verified against, in Docker, by a separate
verification pass after you declare done:

  INVARIANT A (oracle):   run solve.sh, then tests/test.sh  ->  /logs/verifier/reward.txt == 1
  INVARIANT B (null run): run NOTHING, then tests/test.sh   ->  /logs/verifier/reward.txt == 0

If either fails, the build is bounced back to you. See Section 11.

SECTION 2. EXACT DIRECTORY STRUCTURE (NON-MILESTONE — THIS IS THE DEFAULT)
================================================================================

Unless you have a specific reason to build a milestone task, build a NON-MILESTONE task.
number_of_milestones = 0. The layout is exactly this:

  <task-name>/
  |-- task.toml                    REQUIRED   manifest
  |-- instruction.md               REQUIRED   the prompt the agent sees
  |-- environment/
  |   |-- Dockerfile               REQUIRED   (or docker-compose.yaml for multi-container)
  |   |-- .dockerignore            REQUIRED for any non-trivial environment
  |   `-- app/...                  task assets (source, fixtures, docs, data)
  |-- solution/
  |   `-- solve.sh                 REQUIRED   oracle, bash, executable
  |-- tests/
  |   |-- test.sh                  REQUIRED   bash entry point; writes the reward file
  |   |-- test_outputs.py          REQUIRED   pytest assertions
  |   |-- oracle.py                optional but STRONGLY recommended (reference engine)
  |   `-- heldout.py               optional but STRONGLY recommended (fresh-case generator)
  `-- README.md                    OPTIONAL   reviewer notes; never shipped into the image

Rules:

  - The ZIP you produce must contain these entries AT THE ROOT of the archive.
    NO wrapper directory. Unzipping must yield task.toml/instruction.md/environment/... directly.
  - Everything the image builds from lives under environment/. The build context is
    environment/ ONLY. Never reach outside it (no docker-compose `context: ../`).
  - solution/ and tests/ are NEVER copied into the image. They are mounted at runtime.
  - Runtime mount points reserved by the Harbor harness — do not remap or shadow them:
        /tests/            (your tests/ directory)
        /solution/         (your solution/ directory)
        /logs/verifier/    (reward.txt, ctrf.json)
        /logs/artifacts/
  - Do not leave junk in the task root: no jobs/, no stray data/, no CLAUDE.md, no skills.md,
    no AI-scaffolding filenames of any kind. Those are an automatic authenticity flag.

If (and only if) you build a milestone task: there is NO root instruction.md / tests/ /
solution/. Instead steps/milestone_N/{instruction.md, tests/{test.sh,test_mN.py},
solution/{solve.sh,solveN.sh}}, one [[steps]] block per milestone in task.toml,
number_of_milestones must equal the [[steps]] count, [environment] gains workdir, and there
are no top-level [agent]/[verifier] blocks (they become [steps.agent]/[steps.verifier]).
2 to 5 milestones max. Each milestone must be a hard prerequisite of the next.

4.2 THE THREE THINGS THAT GET DOCKERFILES REJECTED

  (a) Unbaked verifier dependencies. There is NO NETWORK at test time. test.sh cannot
      pip install / apt-get install / curl / wget / npm install / cargo fetch anything.
      Therefore EVERY verifier dependency must be baked into the image at build time:

        RUN python3 -m pip install --no-cache-dir --break-system-packages \
                pytest==8.4.1 \
                pytest-json-ctrf==0.3.5 \
                jsonschema==4.23.0

      pytest AND pytest-json-ctrf are mandatory (test.sh emits a CTRF report). Add whatever
      else your test_outputs.py imports. Every version exactly pinned — pinned_dependencies
      is a blocking check. This is the single most common cause of a task that "works locally"
      and dies in CI.

  (b) Solution or tests baked into the image. tests_or_solution_in_image is a blocking check.
      NEVER `COPY tests/ ...` or `COPY solution/ ...` or `COPY ../solution ...`. They are
      mounted at /tests and /solution at runtime. Do not store expected outputs, ground truth,
      or hidden fixtures anywhere the agent can read or write.

  (c) Unpinned anything. Every FROM needs @sha256. Every pip/npm/cargo/gem package needs an
      exact ==version pin. Any downloaded binary needs a version AND a sha256sum -c check:

        ARG OPA_VERSION=0.70.0
        ARG OPA_SHA256=00d114b94fdb1606a48cccdfc73c9ccdc62c38721150131ae578d5ff3df5c084
        RUN curl -fsSL -o /usr/local/bin/opa \
                "https://github.com/open-policy-agent/opa/releases/download/v${OPA_VERSION}/opa_linux_amd64_static" \
            && echo "${OPA_SHA256}  /usr/local/bin/opa" | sha256sum -c - \
            && chmod 0755 /usr/local/bin/opa

      (Build-time downloads are fine. Runtime downloads are not.)

4.3 REQUIRED RUNTIME CONTENTS

  tmux and asciinema MUST be installed in the final image. The agent harness starts a session
  with them. Missing either means EVERY agent run fails with no verifier output at all.

    RUN apt-get update \
        && apt-get install -y --no-install-recommends \
            asciinema \
            ca-certificates \
            tmux \
        && rm -rf /var/lib/apt/lists/*

  One apt transaction per volatility class. Always --no-install-recommends. Always
  rm -rf /var/lib/apt/lists/* IN THE SAME RUN layer. Never apt-get upgrade.

4.4 LAYERING AND HYGIENE (warn-level checks, but reviewers send tasks back for these)

  - Order layers least-volatile to most-volatile: base -> apt -> dependency manifests ->
    dependency install -> source -> build step. WORKDIR /app.
  - Copy manifests before sources (COPY Cargo.toml Cargo.lock first, then src/).
  - Narrow COPY. Never `COPY . /app` unless the context is tiny and .dockerignore is strict.
  - No heredocs in the Dockerfile (`RUN cat > /app/x.py <<'EOF'`). Commit source as files and
    COPY them. check_heredoc_usage flags this.
  - Extract archives at build time and delete the archive in the same layer.
  - No recursive chmod -R / chown -R across /app. Use COPY --chmod=0755 / COPY --chown=.
  - Multi-stage build if you compile artifacts the agent does not need to rebuild. If the
    agent must compile, leaving the toolchain in the final image is fine and expected.
  - Add OpenContainers LABELs (title, description, version, licenses).
  - If the environment needs a pinned upstream repo, clone AND `git checkout <sha>` so the
    agent cannot see later commits containing the answer.

SECTION 5. instruction.md — HOW THE PROMPT MUST SOUND
================================================================================

instruction.md is what the model actually reads. It must read like a real engineer typing into
a terminal agent at 4pm on a Tuesday — not like generated documentation.

The six principles (all High severity; violating any one is a send-back):

  1. CONCISE.        One sentence minimum, three paragraphs maximum. That is the whole budget.
  2. WELL SPECIFIED. The goal is unambiguous. State the WHAT and the output contract.
  3. INTERESTING.    A real developer would find the problem worth solving.
  4. NO ANSWERS, NO HINTS. Requirements yes. Steps, algorithms, thresholds, detection
                     guidance, function signatures — no.
  5. UNIQUE.         Not a re-skin of anything in Terminal-Bench 2/3 or Terminus Edition 1.
  6. ABSOLUTE PATHS. Every path referenced is absolute: /app/policy, /app/out/report.json.
                     Never `config/settings.json`, never `./data`.

Hard prohibitions:

  - No emojis. Not one.
  - Minimal markdown. No #### headers, no nested bullet trees, no bold-marker highlighting of
    "important" values (that is just hint-dropping with extra steps).
  - No canary string. Canary strings are NOT required in Terminus Edition 2 and their presence
    signals you used an obsolete skeleton.
  - No task name / no title comment on the first line.
  - No LLM-tell prose: "You are an expert programmer", "Your goal is to...", "Let's begin!",
    verbose politeness, restating the problem three times.
  - No tool prescriptions you cannot verify ("use vim to edit the file").
  - No vague verbs ("make it better", "fix the issues", "handle errors properly", "optimize").
    Vague is not the same as unhinted. Be precise about the target, silent about the method.

ANTI-EXAMPLES — each of these is a documented rejection:

  BAD (step-by-step walkthrough with solution values):
      "##### Step 3: Optimize the File Sync Utility
       1. Socket Buffer Tuning
          - Set SO_RCVBUF to 262144 bytes (256 KB)
          - Set SO_SNDBUF to 262144 bytes (256 KB)"
      -> You just handed over the answer.

  BAD (hints section describing the answers):
      "##### Detection Guidance
       Look for: currency conversion anomalies that affect only specific currency codes;
       date field values that are valid but transposed relative to the source; ..."
      -> A full taxonomy of the bugs. The task is now trivial.

  BAD (excessive markdown / API-doc styling):
      "##### `revenue-by-category` `GET`
       Array sorted by `totalRevenue` desc. Fields: `category`, `totalRevenue` ..."
      -> Reads like generated documentation, not a human prompt.

  BAD (overly prescriptive):
      "`backend/processor.py` — must export four functions:
         apply_grayscale(input_path: Path, output_path: Path) -> Path ..."
      -> Dictates structure, library, signature. Nothing left to solve.

  BAD (bold markers highlighting solution details):
      "**Policy limit**: $500 USD per single expense.
       **Compliance Rate Definition**: `1.0 - (violation_count / total_expenses)`"
      -> Bolding the exact constants the agent was supposed to derive.

GOOD — this is the actual instruction.md from an accepted task. Study the register:

  "The OPA/Rego bundle in /app/policy is supposed to enforce our WandB model-governance rules,
   but it's misclassifying requests. Right now it waves through GPU inference deployments that
   were never granted capacity, and it blocks production rollbacks that should go straight
   through during an incident. Those are just the two cases we happened to catch — the bundle
   is meant to implement the entire governance review at /app/docs/governance_review.md (the
   current baseline is policy v4.2), and I don't trust that the rest of it is correct either.

   Go through the review and fix the policies so every action it covers gets the decision the
   review actually specifies: model registry promotions, Launch job overrides, and artifact
   alias operations, including how time-bound governance exceptions and their expiry are
   evaluated. The request fixtures we've been testing against are in /app/requests.

   When the policy is right, run /app/run_audit.sh to regenerate the audit report at
   /app/out/audit_report.json and confirm it passes the Cargo-based JSON Schema validator the
   pipeline runs. A report that is schema-valid but contains wrong decisions is not done — the
   decisions have to match the policy in the review."

Notice what it does: names two symptoms out of five real defects; names the authoritative
document but not the rules; names every absolute path the agent needs; states the output
contract; explicitly closes the "schema-valid means done" false-green escape hatch; sounds
irritated and human. Three paragraphs. No markdown. Copy this energy, not this task.

Also: vary your voice between tasks. Some engineers are terse, some are chatty. Do not
converge on a house style — the dataset is trying to capture real variance.

SECTION 6. tests/ — THE VERIFIER
================================================================================

6.1 tests/test.sh — CANONICAL. Reproduce this EXACTLY.

#!/bin/bash

# Verifier dependencies are baked into environment/Dockerfile so the run is offline.
# Add task-specific verifier-only Python packages there, not here.
mkdir -p /logs/verifier

python -m pytest -o cache_dir=/tmp/pytest_cache \
  --ctrf /logs/verifier/ctrf.json /tests/test_outputs.py -rA
RC=$?
if [ "$RC" -eq 0 ]; then
  echo 1 > /logs/verifier/reward.txt
else
  echo 0 > /logs/verifier/reward.txt
fi

Non-negotiable details, each of which is a REAL documented rejection:

  - NO comment and NO blank line between `RC=$?` and the `if`. The validator matches the
    reward block as a contiguous final block. A comment there gets you blocked.
  - NO trailing `exit "$RC"` (or any other command) after the `fi`. The reward block must be
    the LAST thing in the file. A trailing exit gets you blocked.
  - NO `set -euo pipefail` in test.sh. It would abort the script the instant pytest fails,
    reward.txt would never be written, and the harness raises RewardNotFoundError.
  - The reward is BINARY: exactly `1` or exactly `0`. Never a partial score, never a fraction
    of tests passed.
  - reward.txt is written on BOTH paths. The verifier must never exit before assigning reward.
  - No network calls of any kind in test.sh. check_offline_tests and check_test_sh enforce it.
  - test.sh is a bash entry point ONLY. It runs pytest. It does not become a jest/go test/
    cargo test runner. All assertions live in Python.

6.2 tests/test_outputs.py — PYTEST ONLY

  - Python + pytest is mandatory for ALL tasks, regardless of task language. For non-Python
    tasks, write Python tests that invoke the binary / API / service / output files. Need
    browser automation? Use Playwright's PYTHON bindings from pytest, not a JS suite.
  - Module docstring at the top explaining what the file validates. A docstring on EVERY test
    function explaining the behavior being tested. informative_test_docstrings is an LLM-judge
    blocking check.
  - Every requirement in instruction.md maps to at least one test. Anything stated OR implied
    in the prompt and not asserted is a `req-gap` flag. Conversely, do not test behavior the
    instruction never asked for — that is a `phantom-spec` flag.
  - Tests must be independent and order-free. No module-level globals shared between tests.
  - Test BEHAVIOR, not implementation. Never grep the agent's source for `sorted(` or a
    library name. Run the thing and check what it does.
  - No brittle exact-string matching on human-readable output. (Byte-exact checks on a
    machine-readable artifact you specified in the instruction ARE fine and often necessary.)
  - Deterministic. No unseeded randomness, no wall-clock/latency dependence, no flakiness.
    A flaky verifier is a High-severity reject.
  - Same logic for the oracle run and the agent run. No conditional branches that treat them
    differently.
  - Don't sprawl. "Long test files are almost always wrong." Aim for a tight set of tests each
    of which kills a distinct class of wrong solution.
  - Tests must check CORRECTNESS, not just format. Existence + schema-validity alone is a
    rejection. Files exist, schema validates, AND the values are right.

6.3 ANTI-CHEATING — THIS IS SCORED (anti_cheating_measures, hardcoded_solution)

Your task WILL be rejected if the solution can be copied from the test assertions, if the
output format reveals the answer structure, if hardcoded values pass, or if an agent can guess.

Required techniques:

  (a) RE-DERIVE, DON'T ASSERT CONSTANTS.
        BAD:  assert report["total"] == 42
        GOOD: expected = sum(item["value"] for item in json.load(open("/app/data/input.json")))
              assert report["total"] == expected
      The test recomputes ground truth from the inputs. An agent that writes 42 into the file
      still fails when the recomputation disagrees.

  (b) SHIP A REFERENCE ENGINE IN tests/oracle.py.
      A self-contained Python implementation of the ACTUAL rules, living only in the verifier
      (never in the image). Tests compare the agent's artifact/policy/binary against
      oracle.decide(...) rather than against literals. This is how the accepted task does it.

  (c) SHIP A HELD-OUT GENERATOR IN tests/heldout.py.
      A deterministic seeded generator of FRESH cases that are NOT present anywhere in the
      container. At test time, generate N of them (the accepted task uses N=250, seed=12345),
      run the agent's artifact on them, and compare to oracle.py. An agent that special-cases
      the shipped fixtures — or hardcodes the report — cannot pass this. Make the generator
      cover the full input space including the nasty corners.

  (d) RE-EXECUTE THE AGENT'S ARTIFACT, DON'T TRUST ITS OUTPUT.
      If the agent was asked to fix a policy/parser/service, don't just read the report it
      produced — re-run the policy/parser/service yourself from the test and check the
      decisions. Trusting the produced file is the classic hole.

  (e) KEEP A VERIFIER-OWNED COPY OF ANY SCHEMA/SPEC THE TEST DEPENDS ON.
      The accepted task ships tests/audit_report.schema.json, a private copy of
      /app/schema/audit_report.schema.json. Weakening the in-container schema buys the agent
      nothing. Never let the verifier derive ground truth from a file the agent can edit.

  (f) IDEMPOTENCY / DETERMINISM CHECK.
      Re-run the agent's pipeline and assert the output is byte-identical. Catches
      nondeterministic dict ordering, timestamps in output, unstable sorts.

  (g) A NEGATIVE-CONTROL TRAP.
      Build in at least one way for the artifact to be structurally valid but semantically
      wrong, and test the semantics. In the accepted task the report ALWAYS passes the Cargo
      schema validator even with completely wrong decisions — so an agent that stops at
      "schema valid, done" gets reward 0. This directly punishes weakness #24 and #19.

  Also: pytest inserts the test file's directory on sys.path, so `import oracle` and
  `import heldout` from /tests/test_outputs.py work without packaging. Use it.

6.4 RUFF
  All Python in the submission (tests, any helper scripts) must pass ruff. A single unused
  variable (F841) is a documented blocking failure. Do not ship tools/ or scratch scripts in
  the zip — they get linted too.

SECTION 8. HOW TO MAKE IT ACTUALLY HARD
================================================================================

This is the section that decides whether the task is accepted. A structurally perfect task
that Opus solves 90% of the time is worthless. You are engineering a failure.

The rule of thumb: make the task hard because the PROBLEM is deep, not because the INSTRUCTION
is long. Then aim the depth at a documented model weakness. Below are the levers, each mapped
to the weakness catalogue.

8.1 THE UNDER-SPECIFIED-SPEC TRAP (targets Opus #16 "instruction compliance over adversarial
    checking", #19 "under-testing its own solutions", GPT #18 "literalness gap")

    !!! THE PREVIOUS VERSION OF THIS SECTION IS WHY OUR FIRST TWO SUBMISSIONS WERE REJECTED. !!!

    It said: "Name N observed symptoms in the instruction. Plant N+K real defects... The accepted
    task names 2 symptoms and plants 5 defects. This one lever alone does most of the work."

    That is a DEFECT HUNT. A defect hunt classifies as occupation "Software Developers" or
    "Software Quality Assurance Analysts and Testers", and as category "debugging" or
    "software-engineering". EVERY ONE OF THOSE IS BLOCKED. The task is rejected before anyone
    reads it. Do not plant defects. Do not ask the agent to find anything that is "wrong".

    THE LEVER, REBUILT — same psychology, legal premise:

    Nothing in the environment is ever IN ERROR. The shipped values are legitimately correct for
    a PRIOR version, policy or configuration, and they are labelled as such. The agent's job is to
    DERIVE the CURRENT definition from the shipped specification and BUILD the current artifact.

    Name N requirements of the current spec in the instruction. The specification contains N+K.
    Say plainly that the instruction is a summary and the specification is authoritative. A
    literal-compliance model implements exactly the N it was told, self-certifies, and fails the K
    it never went looking for — the identical failure mode, with none of the blocked framing.

    Make the K unnamed requirements SUBTLE and SPEC-DERIVED: a validity window the current tariff
    adds, a member of a protected set the new policy includes, a comparison the revised rule makes
    on two fields where the old one used one.

    The agent is not hunting your mistakes. It is implementing a specification you gave it, and
    the specification is bigger than the summary. That is the whole trick, and it is legal.

8.2 SPEC-VS-IMPLEMENTATION CONFLICT (targets GPT #7 "external documentation verification gap",
    GPT #26 "overconfidence on ambiguous data", Opus #18 "long-context detail distraction")
    Ship an authoritative spec AND an implementation that disagrees with it. Make the
    implementation *plausible* — it should look like something a competent engineer wrote.
    Models trust the code they can see over the document they must read. The instruction must
    make the document authoritative in one sentence and then say nothing more about it.

8.3 SUPERSEDED-VALUE RED HERRINGS / TIE-BREAK RULES BURIED DEEP
    (targets Opus #18, GPT #1 "long-context degradation", GPT #3 "instruction drift")
    Inside a long document, include a clearly-marked HISTORICAL section listing OLD values for
    the very constants the task turns on — a 14-day window that is now 30 days, a 4-GPU cap
    that is now 8, a gate that used to apply only to production and now applies to every stage.
    Mark them honestly ("HISTORICAL — do not apply superseded values") so the task is fair, and
    scatter the CURRENT values across several distant sections so they cannot be grepped in
    one pass. A model that keyword-searches "exception window" hits the wrong number first.
    Put the decision-precedence / tie-break rule in a late section that only matters when two
    rules fire at once — and then make sure some test cases fire two rules at once.

8.4 CROSS-DOCUMENT / CROSS-FILE STATE CONSISTENCY (targets GPT #22 "data consistency across
    related outputs", Opus #17 "exact output contract compliance")
    Require correctness that spans artifacts: derived totals in a summary must agree with the
    per-item rows; ordering must be stable across files; IDs and timestamps must reconcile
    between two outputs. Then TEST the reconciliation, not just each file in isolation. Both
    models fix one output correctly and forget the others.

8.5 EXACT OUTPUT CONTRACT (targets Opus #17, GPT #5 "inconsistency in structured workflows",
    GPT #23 "parser edge cases")
    Deterministic, byte-checkable output. Stable sort keys with a documented tie-break. Sorted
    key order. Exact decimal precision. Newline at EOF. Unknown-field preservation. Consistent
    UTC conversion across a DST boundary. Specify the contract precisely in the instruction
    (this is a requirement, not a hint) and then assert it byte-exactly.

8.6 LONG-CONTEXT NEEDLE RETRIEVAL UNDER DISTRACTION (targets GPT #1: 45.4% vs Opus 68.1% at 1M
    — this is the single widest documented gap between the two models)
    If you use the long_context subcategory you MUST clear this bar:
      - >= 50k tokens of genuine document-like prose SHIPPED with the task (the accepted task
        ships ~63k words / ~3,100 lines of governance review).
      - NOT JSON/JSONL/CSV/TSV/DB-dumps/uniform logs. Not filler, not repeated boilerplate,
        not fifty tiny files.
      - The document must be AUTHORITATIVE to the solution — the oracle is derived from it.
      - It must NOT be solvable by grep, keyword search, field extraction, or top-k stats.
      - Rules must INTERACT across distant sections, so a single extracted sentence gives the
        wrong answer.
      - The instruction says where the documents live without leaking which section matters.
    Good shapes: a long policy/underwriting manual; an engineering standard consolidated from
    four prior memos with a version history; a multi-source investigation corpus (emails,
    chats, tickets, HR records) that must be reconciled.

8.7 CROSS-TOOL / MULTI-LANGUAGE BOUNDARIES (targets Opus #20 "failures occur at the interaction
    boundary between tools", GPT #20 "uncommon languages and frameworks")
    Deliberately span the boundary: shell pipeline + a policy DSL + a compiled binary + a JSON
    schema. The accepted task is Rego + Rust/Cargo + bash + jq. Failures cluster at the seams,
    and the seams are exactly what a single-language benchmark never tests. Bias toward
    languages/tools where the models are documented as weaker (Go, Rust, Rego, niche DSLs)
    rather than mainstream Python/JS web work.

8.8 FALSE-GREEN SELF-VERIFICATION BAIT (targets Opus #24 — the marquee weakness)
    Give the environment a check that PASSES even when the work is wrong. A schema validator
    that only checks shape. A build that compiles. A linter that's clean. A test suite in the
    repo that covers only the happy path. The model runs it, sees green, declares victory,
    and scores 0. Your instruction should acknowledge this trap in one sentence (as the
    accepted task does: "A report that is schema-valid but contains wrong decisions is not
    done") so the task remains FAIR while still catching the behavior.

8.9 REWARD-HACK-PROOFING (targets Opus #12 "training-time reward gaming", GPT #24
    "reward-hacking with weak verifiers")
    Assume the model will try to satisfy the verifier without solving the task. Held-out cases
    (6.3c) and re-execution of the artifact (6.3d) are your defense. Also consider: can the
    agent delete or edit the tests? (No — they're mounted, not in the image.) Can it weaken a
    schema? (No — the verifier has its own copy.) Can it find the answer in git history? (Pin
    the checkout.) Can it decompile a binary to extract expected values? (Don't ship one.)

8.10 WHAT NOT TO DO
    - Do not make it hard with volume: 30 requirements, 12 output files, a 900-word prompt.
      Explicitly rejected.
    - Do not make it hard by being obscure or unfair: contradictory requirements, information
      that isn't in the environment, success that depends on luck, unreliable infrastructure.
    - Do not make it hard by hiding the goal. Well-specified and unhinted are compatible; you
      must have both.
    - Do not build a front-end/visualization task (Opus is weak there, but so is the review
      pipeline — ui_building forces a different skeleton and more review surface).
    - Do not clone an existing Terminal-Bench 2/3 or Terminus Ed.1 task. Uniqueness is checked
      by similarity search.

8.11 SANITY CHECK BEFORE YOU COMMIT TO THE IDEA
    - Could a competent engineer solve this in 1-3 hours with the files provided? (If no, it's
      unfair. Set expert_time_estimate_min honestly.)
    - Does a model that does the obvious thing produce something that LOOKS right and IS wrong?
      (If no, it will pass 80% and you have failed.)
    - Is there at least one failure mode that only shows up on inputs the agent has never seen?
      (If no, add heldout.py.)
    - Is every single thing the tests check either stated or clearly implied in instruction.md?
      (If no, you have a phantom-spec.)
