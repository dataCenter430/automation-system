TERMINUS 2ND-EDITION TASK BUILD PLAYBOOK
Authoritative distillation of the Snorkel/Terminus Edition-2 documentation set.
Read this end to end before you touch a file. It is the spec you will be graded against.

You are about to build ONE complete Terminus Edition-2 task from scratch. Everything you need
to know about what "acceptable" means is in this document. Follow it literally. Where you are

================================================================================
SECTION 1. THE MANDATE
================================================================================

Build a task that frontier models FAIL.

The acceptance bar:

  - Target pass rate for GPT-5.5 and Claude Opus 4.8: 0% to 20%. That is the "hard" band.
  - A task whose WORST model scores above 80% is REJECTED outright. No appeal.
  - Pass rate must be below 100% against SOTA agents, full stop.
  - The ORACLE (solution/solve.sh) must pass: reward.txt == 1.
  - The NULL / NOP run (agent does nothing) must fail: reward.txt == 0.
  - Difficulty must be earned by genuine engineering depth, NOT by piling on instructions.
    A task that is hard only because it has many fiddly requirements is explicitly rejected:
    "If a task is primarily hard due to a large number of edge cases/requirements that are
    not handled well, it should be rejected." Hard means the PROBLEM is hard.

Difficulty bands, verbatim from Task Requirements.txt (this is the authoritative table):

  Hard    accuracy <= 20% on the best model, OR <= 20% on the worst model
  Medium  20% < accuracy <= 60% on the worst model
  Easy    60% < accuracy <= 80% on the worst model
  >80% on the worst model  ->  NOT ACCEPTED

You are building HARD. Set difficulty = "hard" in task.toml and make the task deserve it.

Two invariants you will be independently verified against, in Docker, by a separate
verification pass after you declare done:

  INVARIANT A (oracle):   run solve.sh, then tests/test.sh  ->  /logs/verifier/reward.txt == 1
  INVARIANT B (null run): run NOTHING, then tests/test.sh   ->  /logs/verifier/reward.txt == 0

If either fails, the build is bounced back to you.

SECTION 2. OCCUPATION & SECTOR -- THE CLASSIFIER YOU CANNOT SEE
================================================================================

RULE: OCCUPATION & SECTOR

This is the FIRST decision of a build, before a single file exists.

Three values in the category enum are BLOCKED by a separate LLM classifier that reads your
instruction.md and the source files you ship under environment/. The classifier never reads
task.toml. It predicts the occupational domain of the task independently.

BLOCKED CATEGORIES (reject regardless of task.toml):

  software-engineering
  debugging
  data-processing

Setting a legal enum value in task.toml does NOT make the task legal if the classifier infers
one of the blocked categories from the content. You cannot route around the block by labeling
a code-debugging task as "system-administration". The classifier reads the actual artifacts.

OPEN CATEGORIES (use one of these):

  system-administration
  build-and-dependency-management
  games
  machine-learning
  security
  scientific-computing

To land in an open category, the task DOMAIN must live there. Example themes that pass:

  system-administration   configuring firewall rules, provisioning infrastructure, cron jobs,
                          log rotation, user management, service orchestration
  build-and-dependency-management   crafting Makefiles, resolving dependency conflicts,
                          cross-compiling, package publishing, version pinning pipelines
  games                   building a game engine component, physics simulation, game AI,
                          procedural generation
  machine-learning        hyperparameter tuning pipelines, model serving infra, dataset curation
  security                policy authoring, vulnerability remediation, secret management,
                          compliance verification, audit log forensics
  scientific-computing    numerical simulation, solver implementation, data assimilation,
                          signal processing pipelines

A task might still involve writing code, but if the PROBLEM being solved is "fix the NGINX
config so it passes a compliance audit", that is system-administration. If the problem is
"debug this code that doesn't work", that is debugging and will be blocked.

Choose your sector BEFORE you begin. If you cannot articulate a domain outside software-
engineering / debugging / data-processing, stop and pick a different task idea.

SECTION 3. EXACT DIRECTORY STRUCTURE
================================================================================

Unless you have a specific reason to build a milestone task, build a NON-MILESTONE task.
number_of_milestones = 0. The layout is exactly this:

  <task-name>/
  |-- task.toml                    REQUIRED   manifest
  |-- instruction.md               REQUIRED   the prompt the agent sees
  |-- environment/
  |   |-- Dockerfile               REQUIRED   (or docker-compose.yaml for multi-container)
  |   |-- .dockerignore            REQUIRED for any non-trivial environment
  |   `-- app/...                  task assets (source, fixtures, docs, data)
  |-- solution/
  |   `-- solve.sh                 REQUIRED   oracle, bash, executable
  |-- tests/
  |   |-- test.sh                  REQUIRED   bash entry point; writes the reward file
  |   |-- test_outputs.py          REQUIRED   pytest assertions
  |   |-- oracle.py                optional but STRONGLY recommended (reference engine)
  |   `-- heldout.py               optional but STRONGLY recommended (fresh-case generator)
  `-- README.md                    OPTIONAL   reviewer notes; never shipped into the image

Rules:

  - The ZIP you produce must contain these entries AT THE ROOT of the archive.
    NO wrapper directory. Unzipping must yield task.toml/instruction.md/environment/... directly.
  - Everything the image builds from lives under environment/. The build context is
    environment/ ONLY. Never reach outside it (no docker-compose `context: ../`).
  - solution/ and tests/ are NEVER copied into the image. They are mounted at runtime.
  - Runtime mount points reserved by the Harbor harness -- do not remap or shadow them:
        /tests/            (your tests/ directory)
        /solution/         (your solution/ directory)
        /logs/verifier/    (reward.txt, ctrf.json)
        /logs/artifacts/
  - Do not leave junk in the task root: no jobs/, no stray data/, no CLAUDE.md, no skills.md,
    no AI-scaffolding filenames of any kind. Those are an automatic authenticity flag.

MILESTONE TASKS (only if you have a specific reason):

If (and only if) you build a milestone task: there is NO root instruction.md / tests/ /
solution/. Instead use steps/milestone_N/{instruction.md, tests/{test.sh,test_mN.py},
solution/{solve.sh,solveN.sh}}, one [[steps]] block per milestone in task.toml,
number_of_milestones must equal the [[steps]] count, [environment] gains workdir, and there
are no top-level [agent]/[verifier] blocks (they become [steps.agent]/[steps.verifier]).
2 to 5 milestones max. Each milestone must be a hard prerequisite of the next. Files persist
across milestones. The container filesystem is shared between milestones.

Milestone directory layout:

  <task-name>/
  |-- task.toml                    with [[steps]] blocks, number_of_milestones > 0
  |-- environment/
  |   |-- Dockerfile
  |   `-- ...
  `-- steps/
      |-- milestone_1/
      |   |-- instruction.md
      |   |-- tests/
      |   |   |-- test.sh
      |   |   `-- test_m1.py
      |   `-- solution/
      |       |-- solve.sh         wrapper that calls solve1.sh
      |       `-- solve1.sh        actual oracle for milestone 1 ONLY
      |-- milestone_2/
      |   `-- ...
      `-- milestone_3/
          `-- ...

RULE: CODEBASE_SIZE

This rule is enforced by the validate_task_fields CI check, which is BLOCKING.

codebase_size is a categorical label (not a literal file count); count environment files the
agent works with, not outputs the agent produces. Verbatim from Task Requirements.txt:

  "This is a categorical label (not a literal file count); count environment files the
  agent works with, not outputs the agent produces."

Bands:

  minimal    0-20 files
  small      20+ files
  large      200+ files

Counting procedure: count the files actually shipped under environment/ after .dockerignore
exclusions, then cross-check the declared band.

  find environment -type f | grep -v Dockerfile | wc -l

If the count is 6, declare "minimal". Copying "small" from a template onto a 6-file
environment is a real rejection we have taken. Do not guess.


difficulty -- "hard" (you are building hard). Other legal values: "medium", "easy".

languages -- the languages of the oracle solution / required of the agent. Do NOT add "python"
just because your verifier tests are pytest. Verifier language is not task language.

tags -- 3 to 6 free-form keywords naming tools/libraries/techniques. If you used tool_specific
/ api_integration / db_interaction, the specific tool/framework/database MUST appear in tags.

HARD LIMITS:

  [agent] timeout_sec         must be between 1 and 1800 seconds. 1800 is the ceiling.
  [verifier] timeout_sec      keep realistic; 450-600 is the normal range.
  [environment] allow_internet  MUST be false. This is a blocking static check. There is NO
                              NETWORK at agent time or at test time.
  [environment] build_timeout_sec  bounded; 600-1200 depending on compile weight.

RULE: LOCKFILES

Dependencies are hash-locked. This rule belongs in the DOCKERFILE section where a builder
looks -- not buried in a checklist.

Python installs must use a hash-locked requirements file including transitive deps, installed
with:

  pip install --require-hashes --no-deps -r <lockfile>

A bare `pip install pkg==1.2.3` pin is NOT sufficient and gets flagged.

Lockfile table, verbatim from snorkel_canonical_image_best_practices.md:

  Language    Lockfile
  Python      requirements.txt with exact versions / uv.lock / poetry.lock
  Node        package-lock.json / pnpm-lock.yaml / yarn.lock
  Rust        Cargo.lock
  Go          go.mod and go.sum
  JVM         Pinned Maven or Gradle dependency locks where possible

Reproducible-build scanner: no `go build` in the runtime image; declare and resolve modules
(`go mod edit` + `go mod tidy`) instead of fetching in a way that trips check_reproducible_builds.

Pin downloaded binaries by version AND checksum:

  ARG OPA_VERSION=0.70.0
  ARG OPA_SHA256=00d114b94fdb1606a48cccdfc73c9ccdc62c38721150131ae578d5ff3df5c084
  RUN curl -fsSL -o /usr/local/bin/opa \
          "https://github.com/open-policy-agent/opa/releases/download/v${OPA_VERSION}/opa_linux_amd64_static" \
      && echo "${OPA_SHA256}  /usr/local/bin/opa" | sha256sum -c - \
      && chmod 0755 /usr/local/bin/opa

Avoid curl | sh unless the artifact is version-pinned and checksum-verified.


--- 5.2 THINGS THAT GET DOCKERFILES REJECTED ---

  (a) Unbaked verifier dependencies. There is NO NETWORK at test time. test.sh cannot
      pip install / apt-get install / curl / wget / npm install / cargo fetch anything.
      EVERY verifier dependency must be baked into the image at build time:

        RUN python3 -m pip install --no-cache-dir --break-system-packages \
                pytest==8.4.1 \
                pytest-json-ctrf==0.3.5 \
                jsonschema==4.23.0

      pytest AND pytest-json-ctrf are mandatory (test.sh emits a CTRF report).

  (b) Solution or tests baked into the image. tests_or_solution_in_image is a blocking check.
      NEVER `COPY tests/ ...` or `COPY solution/ ...`. They are mounted at /tests and
      /solution at runtime.

  (c) Unpinned anything. Every FROM needs @sha256. Every pip/npm/cargo/gem package needs an
      exact ==version pin.


--- 5.3 REQUIRED RUNTIME CONTENTS ---

tmux and asciinema MUST be installed in the final image. The agent harness starts a session
with them. Missing either means EVERY agent run fails with no verifier output at all.

  RUN apt-get update \
      && apt-get install -y --no-install-recommends \
          asciinema \
          ca-certificates \
          tmux \
      && rm -rf /var/lib/apt/lists/*

One apt transaction per volatility class. Always --no-install-recommends. Always
rm -rf /var/lib/apt/lists/* IN THE SAME RUN layer. Never apt-get upgrade.


--- 5.4 LAYERING AND HYGIENE ---

  - Order layers least-volatile to most-volatile: base -> apt -> dependency manifests ->
    dependency install -> source -> build step. WORKDIR /app.
  - Copy manifests before sources (COPY Cargo.toml Cargo.lock first, then src/).
  - Narrow COPY. Never `COPY . /app` unless the context is tiny and .dockerignore is strict.
  - No heredocs in the Dockerfile (`RUN cat > /app/x.py <<'EOF'`). Commit source as files and
    COPY them. check_heredoc_usage flags this.
  - Extract archives at build time and delete the archive in the same layer.
  - No recursive chmod -R / chown -R across /app. Use COPY --chmod=0755 / COPY --chown=.
  - If the environment needs a pinned upstream repo, clone AND `git checkout <sha>` so the
    agent cannot see later commits containing the answer.

SECTION 6. instruction.md -- HOW THE PROMPT MUST SOUND
================================================================================

RULE: INSTRUCTION.MD

instruction.md is what the model actually reads. It must read like a real engineer typing into
a terminal agent -- not like generated documentation.

State WHAT correct behaviour is; NEVER prescribe the build, test or workflow steps that
produce it. Keep only the authoritative behaviour and I/O expectations -- drop formatting-
specific requirements and implementation hints.

The six principles (all High severity; violating any one is a send-back):

  1. CONCISE.        One sentence minimum, three paragraphs maximum. That is the whole budget.
  2. WELL SPECIFIED. The goal is unambiguous. State the WHAT and the output contract.
  3. INTERESTING.    A real developer would find the problem worth solving.
  4. NO ANSWERS, NO HINTS. Requirements yes. Steps, algorithms, thresholds, detection
                     guidance, function signatures -- no.
  5. UNIQUE.         Not a re-skin of anything in Terminal-Bench 2/3 or Terminus Edition 1.
  6. ABSOLUTE PATHS. Every path referenced is absolute: /app/policy, /app/out/report.json.
                     Never `config/settings.json`, never `./data`.

RATING: BLOCKING. Review Checklist.md rates hints and STEPWISE INSTRUCTIONS as High severity.
The Blocking Issues doc lists the instruction.md rewrite among the blockers.

Contrast from instruciton_guide.txt:

  BAD:   "First, use the `ls` command to see files, then..."
  GOOD:  "The source data is in /data. Output the result to..."

Hard prohibitions:

  - No emojis. Not one.
  - Minimal markdown. No #### headers, no nested bullet trees, no bold-marker highlighting.
  - No canary string. Canary strings are NOT required in Terminus Edition 2.
  - No task name / no title comment on the first line.
  - No LLM-tell prose: "You are an expert programmer", "Your goal is to...", "Let's begin!".
  - No tool prescriptions you cannot verify ("use vim to edit the file").
  - No vague verbs ("make it better", "fix the issues", "handle errors properly", "optimize").

The same ban on LLM-tells extends to the environment fixtures: they must read like real
internal artefacts -- no robotic meta-commentary, no 'Determination: ...' summaries, no note
column that hands the agent the conclusion it was supposed to derive. Fixtures should not
cite raw input filenames, use explicit AI personas, or have AI-style blue coloring.

ANTI-EXAMPLES (documented rejections):

  BAD (step-by-step walkthrough with solution values):
      "##### Step 3: Optimize the File Sync Utility
       1. Socket Buffer Tuning
          - Set SO_RCVBUF to 262144 bytes (256 KB)
          - Set SO_SNDBUF to 262144 bytes (256 KB)"
      -> You just handed over the answer.

  BAD (hints section describing the answers):
      "##### Detection Guidance
       Look for: currency conversion anomalies that affect only specific currency codes..."
      -> A full taxonomy of the bugs. The task is now trivial.

  BAD (excessive markdown / API-doc styling):
      "##### `revenue-by-category` `GET`
       Array sorted by `totalRevenue` desc. Fields: `category`, `totalRevenue` ..."
      -> Reads like generated documentation, not a human prompt.

  BAD (overly prescriptive):
      "`backend/processor.py` -- must export four functions:
         apply_grayscale(input_path: Path, output_path: Path) -> Path ..."
      -> Dictates structure, library, signature. Nothing left to solve.

Vary your voice between tasks. Some engineers are terse, some are chatty. Do not converge on
a house style -- the dataset is trying to capture real variance.

SECTION 7. tests/ -- THE VERIFIER
================================================================================

RULE: RUFF

EVERY Python file shipped in the submission is linted, not just the tests -- a single unused
variable (F841) in a helper script is a documented BLOCKING failure. Do not ship tools/ or
scratch scripts in the zip; they get linted too.

Make it executable rather than aspirational: the builder RUNS `ruff check` over the task
directory and it must come back clean before the task is declared done.


--- 7.1 tests/test.sh -- CANONICAL ---

Reproduce this EXACTLY:

#!/bin/bash

# Verifier dependencies are baked into environment/Dockerfile so the run is offline.
mkdir -p /logs/verifier

python -m pytest -o cache_dir=/tmp/pytest_cache \
  --ctrf /logs/verifier/ctrf.json /tests/test_outputs.py -rA
RC=$?
if [ "$RC" -eq 0 ]; then
  echo 1 > /logs/verifier/reward.txt
else
  echo 0 > /logs/verifier/reward.txt
fi

Non-negotiable details, each of which is a REAL documented rejection:

  - NO comment and NO blank line between `RC=$?` and the `if`. The validator matches the
    reward block as a contiguous final block. A comment there gets you blocked.
  - NO trailing `exit "$RC"` (or any other command) after the `fi`. The reward block must be
    the LAST thing in the file. A trailing exit gets you blocked.
  - NO `set -euo pipefail` in test.sh. It would abort the script the instant pytest fails,
    reward.txt would never be written, and the harness raises RewardNotFoundError.
  - The reward is BINARY: exactly `1` or exactly `0`. Never a partial score.
  - reward.txt is written on BOTH paths. The verifier must never exit before assigning reward.
  - No network calls of any kind in test.sh. check_offline_tests enforces it.
  - test.sh is a bash entry point ONLY. It runs pytest. All assertions live in Python.


--- 7.2 tests/test_outputs.py -- PYTEST ONLY ---

  - Python + pytest is mandatory for ALL tasks, regardless of task language. For non-Python
    tasks, write Python tests that invoke the binary / API / service / output files.
  - Module docstring at the top explaining what the file validates. A docstring on EVERY test
    function explaining the behavior being tested. informative_test_docstrings is a blocking
    LLM-judge check.
  - Every requirement in instruction.md maps to at least one test.
  - Tests must be independent and order-free. No module-level globals shared between tests.
  - Test BEHAVIOR, not implementation. Never grep the agent's source for library names.
  - No brittle exact-string matching on human-readable output.
  - Deterministic. No unseeded randomness, no wall-clock dependence, no flakiness.
  - Same logic for the oracle run and the agent run. No conditional branches.
  - Tests must check CORRECTNESS, not just format. Existence + schema-validity alone is a
    rejection.


--- 7.3 ANTI-CHEATING (scored by anti_cheating_measures, hardcoded_solution checks) ---

Your task WILL be rejected if the solution can be copied from the test assertions.

Required techniques:

  (a) RE-DERIVE, DON'T ASSERT CONSTANTS.
        BAD:  assert report["total"] == 42
        GOOD: expected = sum(item["value"] for item in json.load(open("/app/data/input.json")))
              assert report["total"] == expected

  (b) SHIP A REFERENCE ENGINE IN tests/oracle.py.
      A self-contained Python implementation of the ACTUAL rules, living only in the verifier.
      Tests compare the agent's artifact against oracle.decide(...) rather than literals.

  (c) SHIP A HELD-OUT GENERATOR IN tests/heldout.py.
      A deterministic seeded generator of FRESH cases that are NOT present anywhere in the
      container. At test time, generate N of them, run the agent's artifact on them, and
      compare to oracle.py.

  (d) RE-EXECUTE THE AGENT'S ARTIFACT, DON'T TRUST ITS OUTPUT.
      If the agent was asked to fix a policy/parser/service, re-run it yourself from the test.

  (e) KEEP A VERIFIER-OWNED COPY OF ANY SCHEMA/SPEC THE TEST DEPENDS ON.
      Weakening the in-container schema buys the agent nothing.

  (f) IDEMPOTENCY / DETERMINISM CHECK.
      Re-run the agent's pipeline and assert the output is byte-identical.

  (g) A NEGATIVE-CONTROL TRAP.
      Build in at least one way for the artifact to be structurally valid but semantically
      wrong, and test the semantics.

SECTION 8. solution/solve.sh -- THE ORACLE
================================================================================

  #!/bin/bash
  set -euo pipefail

  # ... solution commands ...

Requirements:

  - Bash. Starts with `#!/bin/bash` and `set -euo pipefail` (unlike test.sh, where it is
    forbidden).
  - HUMAN-WRITTEN in style: clear, commented, purposeful.
  - DERIVES the answer. It must be a real implementation, not a script that echoes the
    expected outputs into place. `echo "42" > /output/result.txt` is an instant reject.
  - DETERMINISTIC. Same bytes every run. Seed any randomness. `ls | sort`, not bare `ls`.
  - IDEMPOTENT. Running it twice must be safe and produce the same state.
  - SELF-CONTAINED. No external deps, no network, no downloads (allow_internet = false).
  - It must end by producing exactly the artifacts instruction.md promised, at the exact
    absolute paths.


================================================================================
SECTION 9. HOW TO MAKE IT ACTUALLY HARD
================================================================================

This section decides whether the task is accepted. A structurally perfect task that Opus
solves 90% of the time is worthless. You are engineering a failure.

The rule of thumb: make the task hard because the PROBLEM is deep, not because the INSTRUCTION
is long. Then aim the depth at a documented model weakness.

9.1 THE SPEC-IMPLEMENTATION CONFLICT
Ship an authoritative spec AND an implementation that disagrees with it. Make the
implementation plausible. Models trust code over documents.

9.2 SUPERSEDED-VALUE RED HERRINGS
Inside a long document, include a clearly-marked HISTORICAL section listing OLD values.
Mark them honestly so the task is fair. Scatter CURRENT values across distant sections.

9.3 CROSS-DOCUMENT STATE CONSISTENCY
Require correctness that spans artifacts: derived totals must agree with per-item rows;
ordering must be stable across files; IDs must reconcile. Test the reconciliation.

9.4 EXACT OUTPUT CONTRACT
Deterministic, byte-checkable output. Stable sort keys with a documented tie-break.
Exact decimal precision. Newline at EOF.

9.5 LONG-CONTEXT NEEDLE RETRIEVAL
If you use the long_context subcategory you MUST clear this bar:
  - >= 50k tokens of genuine document-like prose SHIPPED with the task.
  - NOT JSON/JSONL/CSV/TSV/DB-dumps/uniform logs. Not filler, not boilerplate.
  - The document must be AUTHORITATIVE to the solution.
  - It must NOT be solvable by grep, keyword search, or field extraction.
  - Rules must INTERACT across distant sections.
  - The verifier checks behavior or outputs that depend on details from the long documents.
  - If multiple documents are provided, the task requires resolving interactions across
    documents, not just opening one obvious file.
  - The instruction tells the agent where the long documents live without leaking the exact
    answer path.

Immediate rejection patterns for long_context:
  - Under 50k valid long-context tokens.
  - A generator script but no shipped document corpus.
  - Large JSONL/CSV/TSV/database inputs labeled as long context.
  - Uniform machine logs with clear delimiters and fixed fields.
  - Dozens of tiny files whose aggregate content is far below 50k tokens.
  - Long files used only for keyword search or simple analytics.
  - Long documents that are historical, optional, non-authoritative, or unrelated to the oracle.
  - Repeated/filler text used to inflate size.

9.6 CROSS-TOOL / MULTI-LANGUAGE BOUNDARIES
Deliberately span boundaries: shell pipeline + policy DSL + compiled binary + JSON schema.
Bias toward languages where models are weaker (Go, Rust, Rego, niche DSLs).

9.7 FALSE-GREEN SELF-VERIFICATION BAIT
Give the environment a check that PASSES even when the work is wrong. A schema validator
that only checks shape. A build that compiles. The model runs it, sees green, declares
victory, and scores 0.

9.8 DOCUMENTED MODEL WEAKNESSES TO EXPLOIT
Consult the weaknesses documentation. Both models share these failure modes:
  - Miss exact output contracts (ordering, precision, schema fields).
  - Fail at edge cases in parsers and data transforms.
  - Can be gamed by brittle verifiers (design yours to resist gaming).
  - Miss deeper consistency rules across multiple related files or output artifacts.
  - Fail on messy real-world input: duplicate rows, malformed lines, timezone boundaries,
    mixed encodings, empty records, stable sorting requirements, partially corrupted data.

Claude-specific weaknesses:
  - Follows prompts closely but does not proactively question unusual data or hidden
    constraints unless the task structure explicitly requires it.
  - Can overlook small but important details buried across many files in a large codebase.
  - Often produces a good-looking solution without running sufficient negative or edge-case
    checks.
  - Failures tend to occur at the interaction boundary between tools in multi-language tasks.

GPT-specific weaknesses:
  - Long-context degradation: 45.4% at 1M tokens.
  - More likely to "helpfully" deviate from instructions.
  - Tends to construct a "reasonable" interpretation when input data is ambiguous rather than
    flagging ambiguity.
  - Noticeably weaker output on Go, Rust, and niche frameworks.

9.9 WHAT NOT TO DO
  - Do not make it hard with volume: 30 requirements, 12 output files, a 900-word prompt.
  - Do not make it hard by being obscure or unfair.
  - Do not build a front-end/visualization task.
  - Do not clone an existing task.
  - Do not hand the agent the answer in environment fixtures (e.g., a Notes column that says
    which product "does NOT qualify" or "qualifies at the 8-year credit rate").

SECTION 10. REFERENCE IMPLEMENTATION
================================================================================

What an accepted task looks like end to end:

  ENVIRONMENT: A long authoritative document (governance review, policy manual, spec); a
  partially broken implementation; request fixtures; a strict JSON Schema; a validation
  pipeline.

  THE HARDNESS: The instruction names a few symptoms. The implementation has more defects
  than named, which must be reconstructed from the long document -- where superseded values
  sit as bait. A validator passes on structurally valid but semantically wrong output.

  THE VERIFIER: oracle.py is a reference decision engine. heldout.py generates fresh requests
  that exist nowhere in the container. test_outputs.py checks the bundle compiles, runs
  targeted tests where expected values are computed by oracle.decide, evaluates on shipped
  fixtures AND held-out requests, validates against the VERIFIER'S OWN copy of the schema,
  checks decisions/ordering/consistency, re-runs the pipeline and asserts byte-identical
  output.

SECTION 13. DOC CONTRADICTIONS
================================================================================

The source docs contradict each other in places. This section lists each contradiction found,
states which source said what, and gives the single ruling the build session must follow.

1. TEST DEPENDENCIES IN THE IMAGE
   Older onboarding material: test/verifier deps must NOT be baked into the image.
   Task Requirements.txt + allow_internet = false blocking check: ALL verifier deps MUST be
   baked in, because there is no network at test time.
   RULING: Task Requirements wins. BAKE THE DEPS IN. The old rule was about not COPYing the
   tests/ and solution/ DIRECTORIES into the image -- that still stands. Install packages;
   never copy test code.

2. VERIFIER MAY DOWNLOAD DEPS
   Review Checklist: "Verifiers can download dependencies that are needed only for
   verification, but they must use pinned versions."
   RULING: OBSOLETE. allow_internet = false means test.sh has no network at all. Zero
   downloads in test.sh.

3. DIGEST PINNING SEVERITY
   Review Checklist: rates "Base Images versions are pinned" as LOW severity / nice-to-have.
   Task Requirements + Dockerfile best-practices: check_pinned_images is BLOCKING.
   RULING: BLOCKING wins. Digest-pin every FROM. Exact-pin every language package.

4. DIFFICULTY BANDS
   New Task Start Guide: "pass rate should be 0~20%, thus HARD".
   Task Requirements: gives the full table.
   RULING: Use Task Requirements.txt -- they agree; the table is the authority.

5. GO BASE-IMAGE DIGEST
   snorkel_canonical_image_best_practices.md and Dockerfile & Image Best Practices.txt give
   slightly different Go digests (two characters differ).
   RULING: The canonical-image doc is the designated sanctioned list. Verify the digest
   resolves before relying on it if you build a Go task.

6. TIMEOUTS
   The skeleton template shows [agent] timeout_sec = 900 / [verifier] 450; the best-practices
   doc shows 900/420; accepted tasks use 1800/600; milestone guide shows 1200 per step.
   RULING: No single mandated value. [agent] timeout_sec has a HARD RANGE of 1 to 1800
   seconds. Pick realistic values; the oracle must comfortably finish inside them.

7. CANARY STRINGS
   The prompt-styling guide's principle 6 is titled "must use absolute paths" but its body
   talks about canary strings (copy-paste error in source).
   RULING: Both rules are real and both apply -- absolute paths everywhere, and NO canary
   string anywhere in instruction.md.

8. HEREDOCS
   The Dockerfile guidance bans heredocs (check_heredoc_usage).
   Accepted solve.sh uses heredocs freely.
   RULING: The ban applies to the DOCKERFILE only. Heredocs in solution/solve.sh are fine.

9. CATEGORY ENUM vs CLASSIFIER
   task.toml has a category field from a closed vocabulary.
   A separate LLM classifier reads instruction.md and shipped source and predicts category.
   RULING: The classifier reading content wins over the field value. Setting a legal value in
   task.toml does NOT make the task legal if the content classifies into a blocked category.
   See SECTION 2 (RULE: OCCUPATION & SECTOR).

10. CODEBASE_SIZE LABEL
    Some docs describe it as approximate ("rough scale").
    Task Requirements.txt clarifies: "This is a categorical label (not a literal file count);
    count environment files the agent works with, not outputs the agent produces."
    RULING: Count the files shipped under environment/ after .dockerignore, then declare the
    band. The validate_task_fields CI check enforces this.

11. LLM-TELL REJECTION SCOPE
    The prompt-generation project docs (different Snorkel workstream) describe rubric and
    document-format rules that do NOT apply to Terminus tasks.
    RULING: Ignore that project's specific rubric format rules. However, the reviewer
    rejection patterns transfer: LLM-tells in generated artefacts (robotic meta-commentary,
    citing raw input filenames, explicit personas, 'Determination: ...' summaries) and input
    files that are too thin or that hand the agent the answer are BLOCKING in Terminus too.

12. RUBRIC LINE FORMAT
    Rubrics Guideline: every criterion line starts with "Agent" and ends with ", [Score]".
    Some examples show scores without leading space.
    RULING: Start with "Agent", end with ", +N" or ", -N" (comma, space, sign, digit).

13. APT PACKAGE VERSION PINNING
    Review Checklist: "Apt package versions are pinned" rated as LOW severity.
    Dockerfile best-practices: "All dependencies are exact-pinned".
    RULING: Apt packages do NOT require version pins for common packages (the low severity
    holds), but language-level packages (pip, npm, cargo, gem) MUST be exact-pinned. If an
    apt package is niche and likely to change behavior, pin it.
